# LocalScript Interface Specification (Freeze)

## Назначение интерфейса

`LocalScript` — локальный UI для постановки задач, генерации, правки и проверки Lua-скриптов под MWS Octapi через уже существующие API-эндпоинты (`/generate`, `/edit`, `/validate`).

Цель интерфейса:
- сделать основной путь "задача -> код -> проверка -> знания" понятным с первого взгляда;
- дать инженерно прозрачный обзор результата (статус, модель, проходы, проверка, диагностические признаки);
- поддержать демо и защиту без изменения бэкенд-ядра.

## Главный пользовательский сценарий

1. Пользователь формулирует задачу в зоне "Постановка задачи".
2. Выбирает режим работы (`Новый скрипт`, `Правка`, `Проверка`).
3. Запускает выполнение (`Выполнить` / `Сгенерировать по задаче`).
4. Получает код в редакторе (главный артефакт), затем при необходимости проверяет или правит его.
5. В нижней панели анализирует:
   - проверку;
   - использованные знания;
   - пояснение модели;
   - диагностику (в режиме "Глубоко");
   - встроенную справку.

## Режимы работы

### Новый скрипт
- Отправляет постановку задачи в `POST /generate`.
- Код редактора в запрос не подмешивается.
- Используется для первичного создания Lua-скрипта.

### Правка
- Отправляет инструкцию + текущий код редактора в `POST /edit`.
- Используется для итеративного улучшения уже полученного кода.

### Проверка
- Отправляет только код редактора в `POST /validate`.
- Используется для локальной проверки синтаксиса и доменных правил.

## Режимы ответа API

### Быстро
- Компактный ответ API.
- Удобен для быстрого цикла и демо базового сценария.

### Глубоко
- Тот же путь генерации, но в ответ добавляются служебные diagnostics.
- Используется для инженерной прозрачности и разбора на защите.

## Модель доступа

### Гость
- Быстрый вход.
- Заметки/история/черновики сохраняются в `sessionStorage` (до закрытия вкладки).

### Локальный профиль
- Данные сохраняются в `localStorage` браузера.
- Подходит для подготовки к защите и повторных прогонов.

Важно: серверной регистрации/авторизации нет. Интерфейс показывает модель, готовую к будущему расширению (SSO, серверный профиль) без изменения текущего контракта API.

## История, заметки, черновики

- **Заметки**: рабочие тезисы к защите.
- **История**: последние задачи (с возможностью повторной подстановки в поле ввода).
- **Черновики**: сохраненные версии Lua-кода для быстрого возврата.
- Запись истории управляется настройкой "Сохранять историю".

## Справка и блок "Что знает система"

Во вкладке "Справка" доступен компактный справочный блок:
- `wf.vars`;
- `wf.initVariables`;
- различия режимов `Новый скрипт` / `Правка` / `Проверка`;
- различия `Быстро` / `Глубоко`;
- чтение разделов "Ошибки / Предупреждения / Подсказки";
- принципы локальной генерации и ограничения.

## Вкладки результата

Нижняя панель структурирована по вкладкам:
- **Проверка** — статусы валидации, ошибки, предупреждения, подсказки, блок уточнения.
- **Знания** — использованные retrieval chunks.
- **Пояснение** — краткая текстовая интерпретация решения.
- **Диагностика** — метаданные и diagnostics (при наличии).
- **Справка** — встроенная инженерная памятка.

Дополнительно сверху отображается блок "Одним взглядом":
- статус;
- модель;
- число проходов;
- факт второго прохода;
- число чанков знаний;
- сводка проверки.

## Блок "Попробовать пример"

Включает готовые сценарии:
- простой скрипт;
- правка кода;
- запрос на уточнение;
- сложный Octapi-кейс.

Блок используется как быстрый вход и как набор демонстрационных маршрутов для защиты.

## Настройки интерфейса

Диалог "Настройки" (локальное хранение):
- тема (светлая/тёмная);
- масштаб текста интерфейса;
- размер шрифта кода;
- deep по умолчанию;
- авто-открытие вкладки "Диагностика";
- сохранение истории.

## Статусы

Интерфейс явно различает:
- **успех** (`success`) — спокойный зеленый статус;
- **уточнение** (`needs_clarification`) — отдельный выделенный блок с быстрым продолжением;
- **ошибка** (`error`) — красный статус и текст ошибки.

## Пустые состояния

Оформлены отдельные понятные состояния:
- нет истории;
- нет черновиков;
- код еще не создан;
- нет использованных знаний;
- отсутствуют ошибки/предупреждения/подсказки;
- запрос требует уточнения.

## Ограничения интерфейса

- Нет серверной авторизации и многопользовательской совместной работы.
- Нет серверного хранения пользовательского профиля.
- Все пользовательские настройки и профильная память локальны.
- Интерфейс не заменяет эксплуатационную документацию API; он дополняет ее.

## Что показывать на демо

Рекомендуемый короткий маршрут:
1. Показать шапку (статус API/моделей, Гость/Локальный профиль, Быстро/Глубоко).
2. Запустить "Простой скрипт" и показать путь до результата.
3. Открыть вкладки "Проверка", "Знания", "Пояснение".
4. Переключить "Глубоко" и показать вкладку "Диагностика".
5. Запустить сценарий "Уточняющий вопрос" и пройти блок `needs_clarification`.
6. Показать "Настройки" и эффект изменения темы/шрифта кода.

## Freeze-статус

Этот документ фиксирует интерфейс в текущем состоянии. Новые концептуальные UI-фичи в freeze не добавляются; допускаются только точечные исправления дефектов отображения/сборки.
