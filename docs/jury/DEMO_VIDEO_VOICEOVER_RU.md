# Озвучка демо-видео (локальный контур LocalScript)

Текст для записи голосом **отдельно** от экрана. По ходу чтения на видео выполняйте действия в блоке «По видео» каждого раздела.

---

## 0. Вступление (10–20 с)

**Текст:**  
«Это демонстрация решения в **локальном контуре**: на машине запущены Docker Compose, сервис API и локальный **Ollama** с моделями. Внешние облачные API для генерации не используются — только ваш компьютер и контейнеры. Дальше покажу типовой запрос, генерацию **Lua** для Octapi, автоматическую **валидацию**, один цикл **уточнения или правки**, и кратко — требования к железу и ограничениям.»

**По видео:**  
Показать терминал в корне проекта, команду `docker compose up` (или уже поднятый стек + `docker compose ps` с healthy **lua-gen-api**). Кратко проговорить, что это «локальный запуск».

## 2. Локальный контур одной фразой (15 с)

**Текст:**  
«**Контур**: браузер → наш **API** на порту 8000 → **Ollama** в Docker → ответ модели. Код из ответа извлекается, прогоняется **luac** и правилами безопасности; при ошибках возможен второй проход модели — **reflexion**. Параллельно в промпт подмешиваются фрагменты из **базы знаний** — это RAG, не ручной выбор документа.»

**По видео:**  
Переключить режим глубины на **«Глубоко»**, чтобы во время запроса заполнились вкладка **«Диагностика»** и счётчики RAG (опционально открыть настройки и показать, что всё локально — без ключей внешних API).

## 3. Формулировка задачи и реакция системы (30–45 с)

**Текст:**  
«Пользователь формулирует задачу на естественном языке в поле запроса. Нажимаю **Выполнить** в режиме **Новый скрипт**. Система отправляет запрос на сервер, подтягивает релевантные сниппеты знаний, вызывает модель и возвращает **блок Lua** в markdown-ограждении. В чате и в центральном редакторе видно, как система отреагировала: появился код и краткая сводка результата.»

**По видео:**  
Режим **Новый скрипт**. Ввести осмысленный запрос (или **«Попробовать пример → Простой скрипт»**, затем при желании отредактировать текст под «свой» запрос). **Выполнить**. Показать появление ответа в ленте чата и **Lua в редакторе**.

_Пример формулировки (можно вставить в поле):_  
«Сгенерируй Lua-скрипт для Octapi: при старте инициализируй `wf.vars.counter` нулём, при каждом вызове увеличивай на 1 и возвращай строку с текущим значением.»

## 4. Генерация Lua и валидация (25–35 с)

**Текст:**  
«Обязательный элемент — **генерация Lua-кода** под сценарии MWS Octapi. Сразу после ответа срабатывает **валидация**: синтаксис через **luac**, запрет опасных вызовов, проверки согласованности с `wf.vars`. На вкладке **Проверка** видно: пройдена ли проверка, есть ли жёсткие ошибки, предупреждения и подсказки. Это не «красивый текст модели», а машинная проверка артефакта.»

**По видео:**  
Открыть вкладку результата **«Проверка»** — статус, списки ошибок/предупреждений. При необходимости нажать **«Проверить редактор»** в режиме проверки, чтобы явно показать шаг проверки уже готового кода.

---

## 5. Итерация уточнения или исправления (40–60 с)

**Текст:**  
«Требуется показать хотя бы одну **итерацию доработки**. Первый вариант — сценарий **уточнения**: короткий или неоднозначный запрос, система возвращает **needs_clarification** с вопросами; пользователь отвечает в поле быстрого ответа и снова запускает цепочку — получаем обновлённый код. Второй вариант — режим **Правка**: тот же код, новая инструкция, например «добавь проверку на nil» или «верни только число без лишнего текста» — и повторный ответ модели с повторной валидацией. На демо я использую [уточнение / правку — выберите одно и озвучьте].»

**По видео (вариант A — уточнение):**  
Пример **«Уточняющий вопрос»** → **Выполнить** → показать блок уточнения → ввести короткий ответ → продолжить → показать новый Lua и вкладку **Проверка**.

**По видео (вариант B — правка):**  
Переключить **Правка**, ввести инструкцию доработки, **Выполнить**, показать изменённый код и снова **Проверка**.

## 6. Шаблоны, знания, агентный пайплайн (25–35 с)

**Текст:**  
«**Шаблоны и примеры** в интерфейсе задают типовые формулировки — это ускорение демо и реальной работы. **База знаний** используется автоматически: на вкладке **Знания** видны **фрагменты**, попавшие в контекст генерации — retrieval top-k. **Пайплайн**: уточнение при необходимости → RAG → первая генерация → валидация → при сбое синтаксиса **второй запрос** к модели с текстом ошибок — это внутренняя **reflexion**, без отдельного «агента-оркестратора» в UI, но это полноценный цикл проверки и доработки на сервере. В режиме **Глубоко** на вкладке **Диагностика** видны размеры промпта, этапы и признак reflexion.»

**По видео:**  
Вкладка **«Знания»** — карточки чанков. Вкладка **«Диагностика»** (после запроса в режиме **Глубоко**) — `retrieval_chunks_count`, итерации, при наличии — reflexion. Прокрутить блок примеров (**Примеры**), если нужно явно сказать «шаблон».

## 7. Железо, модель, ограничения (25–40 с)

**Текст:**  
«**Железо:** для воспроизведения из коробки достаточно машины с **Docker** и примерно **8–16 ГБ RAM** под модели в CPU-режиме; для ускорения есть профиль с **GPU NVIDIA**. **Модели** по умолчанию — **qwen2.5-coder:7b** с запасным **deepseek-coder:6.7b**; обе тянутся в локальный Ollama. **Ограничения:** один запрос — один нестриминговый вызов генерации; лимиты **num_ctx** и **num_predict** задают длину контекста и ответа; таймаут HTTP защищает от зависаний на медленном CPU. Валидация на чистом Lua может слегка отличаться от рантайма Octapi — это зафиксировано в документации решения. Внешние облачные LLM в этом контуре **не** используются.»

**По видео:**  
По желанию показать `README.md` или `INFO.md` с таблицей моделей / переменных окружения (кратко, 2–3 секунды на экран).

## Синхронизация «видео + голос»

1. Снимите **видео экрана** по шагам 0–2, затем 3–8, без спешки на переключения вкладок.  
2. Запишите **озвучку** по этому файлу в тихой комнате; при монтаже подложите дорожку под видео и подрежьте паузы, где интерфейс догоняет модель.  
3. Если запись идёт одним дублем: читайте короткие абзацы и делайте паузу на длинные ответы модели.
