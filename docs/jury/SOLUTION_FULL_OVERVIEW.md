# LocalScript: Full Solution Overview

## 1. Название решения

**LocalScript** — локальный инженерный ассистент для генерации и проверки Lua-кода под MWS Octapi.

## 2. Короткое описание

LocalScript превращает текстовую задачу пользователя в валидный Lua-код для Octapi и сразу проверяет результат.  
Решение работает локально через Ollama и не использует внешние AI API.  
Качество обеспечивается не только моделью, но и оркестрацией: retrieval, validation, clarification, fallback и управляемый second pass.  
Поверх API есть freeze-ready интерфейс, ориентированный на инженерный workflow и демонстрацию ценности решения на защите.

## 3. Какую проблему решает продукт

- ускоряет написание Lua-скриптов для low-code сценариев Octapi;
- снижает риск синтаксических и доменных ошибок до интеграции;
- делает процесс генерации и правки воспроизводимым и контролируемым;
- упрощает onboarding и ежедневную работу инженеров с типовыми сценариями.

## 4. Для кого сделан

- инженеры и разработчики сценариев MTS / Octapi;
- команды, которым нужен локальный ассистент без передачи данных во внешние LLM-сервисы;
- участники и reviewers хакатона, которым важны воспроизводимость и проверяемые метрики.

## 5. Почему это актуально для MTS / Octapi

- локальность и контроль данных соответствуют корпоративным требованиям;
- архитектура готова к постепенной эволюции без ломки ядра;
- интерфейс оформлен как внутренний инструмент, а не лабораторный прототип;
- есть подтвержденные строгие метрики и честно зафиксированные ограничения среды.

## 6. Что умеет система

- генерировать новый Lua-код из текстовой задачи;
- редактировать существующий Lua-код по инструкции;
- проверять синтаксис и доменные ограничения;
- запрашивать уточнение только когда контекста недостаточно;
- использовать локальную базу знаний для усиления генерации;
- возвращать explanation, диагностику и использованные чанки знаний.

## 7. Пользовательский путь

1. Пользователь формулирует задачу (новый код / правка / проверка).
2. Система определяет режим и при необходимости использует контекст.
3. Orchestrator собирает prompt с retrieval-контекстом.
4. LLM генерирует Lua-код.
5. Validator проверяет результат (`luac` + доменные правила).
6. Пользователь получает код, статус, explanation и диагностику.
7. При неоднозначности система запрашивает clarification и продолжает по ответу пользователя.

## 8. Техническая архитектура (обзор)

Решение построено как управляемый оркестраторный контур:

- `**api/main.py`** — FastAPI API-слой;
- `**api/orchestrator.py**` — маршрутизация и логика pipeline;
- `**api/retriever.py**` — retrieval из локального корпуса;
- `**api/validator.py**` — синтаксис Lua и доменные проверки;
- `**api/prompts.py**` — шаблоны промптов;
- `**api/static/***` — интерфейс LocalScript;
- `**docker-compose*.yml**` — воспроизводимый запуск среды.

## 9. Компоненты системы

- `ollama` — локальный inference backend;
- `api` — orchestration + endpoints;
- `ollama-pull` — автоматическая подготовка моделей;
- knowledge corpus (`api/knowledge/*`, `api/docs_snippets.json`);
- frontend LocalScript для интерактивной работы.

## 10. Модели и их роль

- **Primary:** `qwen2.5-coder:7b` — основной генератор.
- **Fallback:** `deepseek-coder:6.7b` — резерв при инфраструктурных сбоях.

Fallback используется как защитный механизм стабильности, а не как скрытая замена бизнес-логики.

## 11. Как работает retrieval

- BM25/Whoosh по локальному корпусу;
- keyword bonus для релевантности;
- ограниченный `top_k` для контроля размера prompt;
- retrieval-контекст добавляется в системный промпт перед генерацией.

## 12. Как работает validation

- синтаксис через `luac` (Lua 5.5 в образе API);
- доменные проверки (`wf.vars`, `wf.initVariables`, запреты опасных вызовов);
- разделение на `hard_errors`, `warnings`, `hints`.

## 13. Clarification, второй проход, fallback

- **Clarification:** отдельный `needs_clarification` для реально неоднозначных запросов.
- **Second pass (reflexion):** повторная попытка при неуспешной первой валидации.
- **Fallback:** переключение на резервную модель только при транспортных/инфраструктурных ошибках.

## 15. Большой раздел: Фичи решения

Ниже ключевые фичи в формате "что делает / зачем нужна / польза".

### 15.1 Генерация нового Lua-кода

- **Что:** создает скрипт по текстовой задаче.
- **Зачем:** ускорить baseline-разработку сценария.
- **Польза:** быстрый старт и меньше ручной рутины.

### 15.2 Правка существующего кода

- **Что:** изменяет текущий код по инструкции.
- **Зачем:** поддержать iterative workflow.
- **Польза:** экономия времени на доработках.

### 15.3 Проверка кода

- **Что:** отдельная валидация текущего Lua.
- **Зачем:** быстро локализовать ошибки до запуска в контуре.
- **Польза:** снижение интеграционных рисков.

### 15.4 Clarification

- **Что:** запрашивает уточнение при недостатке контекста.
- **Зачем:** не генерировать псевдокорректный код "вслепую".
- **Польза:** выше точность на неоднозначных запросах.

### 15.5 Retrieval

- **Что:** подмешивает релевантные знания.
- **Зачем:** снизить галлюцинации и повысить доменную точность.
- **Польза:** лучшее качество кода на практических кейсах.

### 15.6 Локальная база знаний

- **Что:** корпус чанков и сниппетов в репозитории.
- **Зачем:** работать без внешних хранилищ.
- **Польза:** контроль данных и воспроизводимость.

### 15.7 Fallback-модель

- **Что:** запасной путь при инфра-сбоях.
- **Зачем:** повысить устойчивость ответа.
- **Польза:** меньше срывов пользовательского сценария.

### 15.8 Validation

- **Что:** `luac` + доменные правила.
- **Зачем:** не пропускать опасные/некорректные конструкции.
- **Польза:** предсказуемость качества результата.

### 15.9 Второй проход (reflexion)

- **Что:** дополнительная попытка после провала валидации.
- **Зачем:** auto-repair без ручного цикла.
- **Польза:** выше шанс успешного ответа с первого пользовательского запроса.

### 15.10 Интерфейс LocalScript

- **Что:** единая рабочая поверхность (задача/код/результат).
- **Зачем:** сделать решение usable как внутренний инструмент.
- **Польза:** понятный и быстрый путь для инженера.

### 15.11 Быстро / Глубоко

- **Что:** режимы детализации выдачи.
- **Зачем:** баланс скорости и глубины анализа.
- **Польза:** удобство для разных этапов работы и демо.

### 15.12 Гость / Локальный профиль

- **Что:** легкая модель доступа без серверной авторизации.
- **Зачем:** показать продуктовую зрелость и будущую масштабируемость.
- **Польза:** ясная UX-модель и удобство локальной работы.

### 15.13 Заметки, история, черновики

- **Что:** локальные артефакты сессии.
- **Зачем:** сохранить рабочий контекст.
- **Польза:** ускорение повторных итераций.

### 15.14 Справка и "Что знает система"

- **Что:** встроенная инженерная справка.
- **Зачем:** снизить порог входа и ошибки использования.
- **Польза:** быстрее получить корректный результат.

### 15.15 "Попробовать пример"

- **Что:** демо-сценарии в интерфейсе.
- **Зачем:** быстро показать ценность на защите.
- **Польза:** предсказуемая демонстрация.

### 15.16 Настройки интерфейса

- **Что:** тема, шрифты, диагностика, сохранение истории и т.п.
- **Зачем:** адаптация UI под пользователя.
- **Польза:** комфорт и продуктивность.

### 15.17 Статусы, диагностика, explanation, знания в UI

- **Что:** структурированный результат выполнения.
- **Зачем:** сделать поведение системы прозрачным.
- **Польза:** проще разбирать ошибки и защищать инженерные решения.

## 16. Технические характеристики (подтвержденные)

- **Стек:** Python + FastAPI + JavaScript frontend + Docker Compose.
- **Inference:** локальный Ollama.
- **Модели:** `qwen2.5-coder:7b` + `deepseek-coder:6.7b`.
- **Strict профиль:** через `docker-compose.hackathon.yml`.
- `**num_ctx`:** 4096.
- `**num_predict`:** strict 256 (freeze профиль имеет отдельные дефолты).
- `**OLLAMA_NUM_PARALLEL`:** 1.
- `**RETRIEVAL_TOP_K`:** 4 (текущий baseline).
- `**OLLAMA_HTTP_TIMEOUT_S`:** 240 в актуальном strict/freeze docs-консенсусе.
- **Lua validator:** `luac` (Lua 5.5 в контейнере API).
- **Локальный запуск:** `docker compose ...` из корня.
- **Внешние AI API:** отсутствуют.

## 21. Ограничения и честные оговорки

- часть показателей зависит от стенда (железо, сеть, Docker окружение);
- GPU/VRAM claims подтверждаются только на целевой NVIDIA-машине;
- возможна недетерминированность latency на отдельных тяжелых кейсах;
- retrieval benchmark можно расширить (3/4/5) для дополнительной статистической уверенности;
- server-side память, профили и корпоративная auth — это roadmap, не текущая реализация.

## 22. Что можно развивать потом (post-freeze)

- richer retrieval benchmark на расширенном representative subset;
- дальнейшая чистка retrieval-шума и качество ранжирования;
- разведение ролей моделей для отдельных этапов pipeline;
- серверные профили/память как отдельный этап;
- корпоративная авторизация и shared workspaces;
- расширение корпуса знаний;
- более глубокая диагностика и observability.

## 23. Почему это уже не просто прототип

LocalScript уже выглядит как реальный внутренний продукт, потому что сочетает:

- рабочий инженерный цикл (generate/edit/validate/diagnostics),
- воспроизводимый deployment,
- измеримые strict-результаты,
- целостный интерфейс и документы для передачи,
- честную рамку ограничений и управляемый roadmap.
